Getting started

Install the app, open admin, then wire up projects and SLAs.

  1. Install AutoChase from the Atlassian Marketplace.
  2. Open Jira → Apps → AutoChase (or Manage apps → AutoChase).
  3. Optional: configure Departments, Teams, and Escalation hierarchy.
  4. Create SLA rules, add Projects from Jira, and assign default SLAs / teams as needed.

Sidebar order in the app: Dashboard, Tracked issues, SLA analytics, Projects, SLA rules, Departments, Teams, Escalation hierarchy.

Tracked issues

Live list of issues AutoChase is monitoring, with quick status filters.

KPI summary

At the top: Total, On track, At risk, Breached — counts with the same labels as the dashboard; clicking a card applies that status filter.

Filters

Filter row: All, On track, At risk, Breached, Resolved. When a filter is active, the UI shows how many issues match out of the total tracked set.

Table

Columns: Issue key, Project, Priority, SLA (name of the assigned SLA; a hint may appear if priority timing needs attention), Status, Escalation (level).

If the list is empty, add a project under Projects and create or sync issues in that project.

Projects

Choose which Jira projects AutoChase tracks and how new issues get a default SLA.

Select a project from your Jira instance to track. Assign a default SLA so new issues can pick up that SLA automatically (subject to your SLA matching rules).

Sync and search

  • Search projects — filter the list of projects already added in AutoChase.
  • Sync projects — pulls project metadata from Jira into AutoChase.
  • Sync issues — pulls issues for tracked projects into tracking (bounded per project in the backend).

Add a project

  1. Click Load projects to populate the dropdown from Jira (if needed).
  2. Pick a project in Load projects dropdown.
  3. Enter Owner email (required). Used for escalation when a ticket has no assignee.
  4. Click Add project.

Projects are sourced from the Jira list above; owner email is required for escalation when tickets are unassigned.

Project list actions

Each row shows project name, key, default SLA name, and team. Actions:

  • Edit — opens the project dialog (see below).
  • Assign SLA — set or clear the default SLA for new issues in this project.
  • Assign team — link the project to a team (organizational structure; SLA resolution order is described in that modal).
  • Remove — stop tracking this project in AutoChase.

Edit project (modal)

Fields:

  • Project name
  • Jira project key (e.g. PROJ) — required
  • Description (optional)
  • Project owner email — required; escalation when unassigned
  • Team (optional)
  • Default SLA (optional) — for new issues in this project

Use Assign SLA / Assign team from the list for quick changes without opening the full edit form.

Escalation hierarchy

Upload your org structure so manager-chain visibility and escalations resolve correctly.

Go to the Escalation hierarchy tab in the sidebar.

Step 1: Download the template

Click Download template for a CSV/TSV in the correct format. Columns: employee_id, name, email, reports_to_id.

Step 2: Import the data

  • Paste CSV/TSV into the text area, or
  • Import (choose file) — CSV, TSV, or Excel (.xlsx/.xls).

Apple Numbers files should be exported to Excel or CSV first. After import, data appears in the table; you may see progress such as “Importing…” and “Resolving Jira account IDs…”.

Step 3: Resolve Jira IDs

Click Resolve Jira IDs to fetch Jira account IDs from email addresses (requires read:jira-user). This links rows to Jira users for escalations and watchers.

Step 4: Verify

Confirm hierarchy and account IDs in the table. Use Add employee to add or edit rows manually.

Departments

Organizational buckets for reporting and structure.

Use Departments to model your organisation for reporting and escalation context. Open Departments from the left navigation.

Add a department

Click Add department, then enter Name (required) and Description (optional). Click Save.

Manage

The table lists departments with status. Use Edit or Delete. If a department is deleted, teams linked to it become unassigned.

Best practice

  • Keep names consistent (e.g. Engineering, Support, Product).
  • Create departments before teams when possible.
  • SLA defaults are configured at project/SLA level, not on the department row itself.

Teams

Group projects under departments for clearer analytics and structure.

Open Teams from the left navigation.

Add a team

Click Add team, enter Team name (required) and Department, then Save.

Manage

Edit or Delete from the table. Deleting a team leaves projects unassigned from that team.

How teams are used

  • Link projects to a team from the Projects tab (or project edit flow).
  • Teams support analytics and org context; SLA timing is driven by SLA rules and project defaults.

SLA rules

Define response and resolution targets, reminders, and leadership visibility thresholds.

An SLA rule sets expected timelines and when reminders and leadership visibility should trigger. You can create a Single SLA (same timings for all priorities) or a Priority-based SLA (P1–P5).

Field reference

  • SLA name — Label for this rule.
  • Response time — Max time to first acknowledge/respond.
  • Resolution time — Max time to resolve/close.
  • Warning threshold (%) — When status becomes At risk (e.g. 80%).
  • Acknowledgment reminders — Based on response-time usage (e.g. 25%, 50%, 75%).
  • Progress reminders — Based on resolution-time usage.
  • Progress check interval — How often progress reminders are evaluated.
  • Leadership visibility levels — Thresholds that notify higher levels in the manager chain.
  • Trigger base (Response / Resolution) — Whether a level is computed from response or resolution time.

Mode 1: Single SLA

Use when every ticket shares one response and resolution target.

Step 1: SLA timing

  1. Open SLA rules and click Create SLA.
  2. Enter SLA name.
  3. Select Single SLA.
  4. Set Response time, Resolution time, and Warning threshold (%) (80 is a common default).
  5. Click Next.

Step 2: Acknowledgment reminders

  1. Add up to three reminder percentages (e.g. 25, 50, 75) based on response time.
  2. Edit the acknowledgment message template if needed.
  3. Click Next.

Step 3: Progress reminders

  1. Add up to three percentages from resolution time (e.g. 50, 75).
  2. Set Progress check interval (minutes).
  3. Edit the progress message template if needed.
  4. Click Next.

Step 4: Leadership visibility levels

  1. Choose the number of levels (e.g. 1–6).
  2. For each level: trigger base Response or Resolution, trigger percentage (1–99), and review the time hint.
  3. Update the leadership comment template if required.
  4. Click Next.

Step 5: Review & submit

  1. Review acknowledgment reminders, progress reminders, leadership rules, and combined trigger order.
  2. Click Submit (or Create SLA).

Result: The SLA is saved and can be assigned from the Projects tab or used in matching rules.

Mode 2: Priority-based SLA (P1–P5)

Use when priorities need different response/resolution targets.

Step 1: SLA timing

  1. Click Create SLA, enter SLA name, select Priority-based SLA (P1–P5).
  2. Fill the P1–P5 grid with response and resolution times.
  3. At least one priority row must have both response and resolution.
  4. Click Next.

Priority mapping

AutoChase maps Jira priorities to internal P1–P5. Typical mapping:

  • P1 — Highest / Critical / Blocker
  • P2 — High / Major
  • P3 — Medium
  • P4 — Low / Minor
  • P5 — Lowest / Trivial

Steps 2–5

Acknowledgment reminders, Progress reminders, and Leadership visibility follow the same flow as Single SLA (percentages use the effective response/resolution time for the issue’s priority). Finish with Review & submit.

Result: Priority-based SLA is saved and can be assigned to projects.

Status lifecycle

  • On track — Within SLA limits.
  • At risk — Warning threshold reached.
  • Breach — Resolution threshold crossed.

Recommended practices

  • Start with a small set of SLAs and expand.
  • Use clear names (e.g. “Support – Priority SLA”).
  • Avoid notification fatigue — keep reminder percentages realistic.
  • Keep leadership levels strictly increasing by trigger percentage.
  • Upload hierarchy before relying on manager-chain visibility.

How it works

End-to-end behavior once projects and SLAs are configured.

  • Auto-tracking: New issues in enabled projects are tracked from creation (subject to sync and app rules).
  • SLA assignment: A matching SLA is applied from project defaults and SLA rules (project, issue type, priority).
  • Status: on_track → at_risk (at warning %) → breach at resolution breach.
  • Reminders: Acknowledgment and progress reminders as usage crosses configured thresholds.
  • Escalation / visibility: Scheduled processing applies leadership visibility rules and org hierarchy.
  • Resolution: When an issue reaches a done/resolved state, tracking stops for that issue.

Configure per-SLA escalation and leadership visibility in the SLA wizard; org hierarchy powers manager-chain routing.

Issue panel

Per-issue SLA context inside Jira.

On each tracked issue, open the right-hand SLA Status panel to see status, escalation level, time remaining, acknowledgment state, and to assign an SLA manually when allowed.

Troubleshooting

ProblemWhat to try
Issue not trackedAdd the project in Projects, then create a new issue or run Sync issues.
No SLA assignedCreate an SLA that matches the project (or *), issue type, and priority; set a project default SLA where appropriate.
Panel shows “not tracked”Only issues in added projects are tracked; the issue may pre-date the project being added — create/update or sync.
Escalation not workingVerify hierarchy upload, Jira ID resolution, and SLA leadership levels. Managers need resolvable Jira accounts (email / account ID).
